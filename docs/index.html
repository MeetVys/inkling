the problem

Markdown is a ceiling. Your thinking deserves more.

You write the doc. The numbers go into a static table because markdown can't chart them. The architecture goes into ASCII art because markdown can't render diagrams. The decision tree gets flattened into bullets. The CSV gets pasted as a code block. The doc ships. It gets archived. You never reopen it — because there's nothing to reopen into.

  • Perf reports with raw numbers in a static table when they deserved a chart bound to the source data — edit the CSV, watch the chart move.
  • Architecture docs that describe systems in prose because markdown can't show a clickable diagram with click-to-explain panels.
  • Decision walks flattened into nested bullets because there's no way to actually walk a decision tree in markdown.
  • Learning notes that read like transcripts. No glossary popovers, no inline quiz, no interactive concept map. Just paragraphs.
  • Vision docs that get shared once and never reopened. The format sets the ceiling for how alive the doc can feel — and markdown's ceiling is flat.
  • The alternatives fail differently. Notion and Observable live on someone else's server. Idyll requires authoring a separate markup language. Claude Artifacts disappear with the chat. None of them give you a self-contained, portable, archive-worthy artifact that runs from your filesystem.

the solution

Let the agent build the doc your thinking deserves.

inkling is a Claude Code skill. You describe the document in natural language — no markup, no slash commands. The agent designs the chassis for this specific doc, invents the structural shapes it needs, writes the styles, wires the interactions, parses your data, and renders a folder of HTML + CSS + JS + data you can open offline, archive forever, or share.

Every render must hit five axes. None are negotiable. If any axis falls below threshold, the agent fixes it before declaring done.

01

Interactive

The doc reacts. Click-to-explain on diagrams, decision walks, drawers, focus mode, scroll-spy, glossary popovers. Things markdown literally cannot do.

02

Computational

The doc computes. Inline CSV parsing renders charts. Decision trees walk from source data. Numbers update when the data changes. Bound, not pasted.

03

Aesthetic

Typographic care, layout discipline, restraint. Real serif type for prose, mono for code, a warm-notebook palette inherited across every doc in the kit.

04

Warm

Paper tones, gentle interactions, no chrome-junk. The doc invites reopening — it reads like a notebook page, not a SaaS landing.

05

Originality

Every render must invent at least one structural shape the kit hasn't seen. No template menu — the agent designs what this doc earns. Below 5/10 is a hard fail.
